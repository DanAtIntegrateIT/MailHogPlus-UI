RAG Explained

How MailHog+ calculates email quality and what your team can tell customers when messages underperform.

MailHog+ starts every message at 10.0 and subtracts points for each issue found. The final value is clamped to 0-10 and rounded to one decimal place.
Start score 10.0
Critical issue -2.2 each
Warning -0.9 each
Info hint -0.25 each
Status Score range What it means
Green 8.0 to 10.0 Generally ready to send, with no major quality blockers.
Amber 5.0 to 7.9 Usable, but issues may hurt rendering, trust, or deliverability.
Red 0.0 to 4.9 High risk. Critical problems should be fixed before sending.
Checks that MailHog+ runs
  • Unsafe HTML: flags tags like script, form, iframe, video, audio, object, embed, canvas, and form inputs.
  • Client compatibility: warns on email-unfriendly CSS/layout patterns (flexbox, CSS grid, fixed positioning, and inline SVG).
  • Images and accessibility: checks missing alt text, image-only emails, and image-heavy content with too little real text.
  • Links and security: checks for JavaScript links, placeholder/broken links, and non-HTTPS URLs.
  • Inbox readiness: checks for preheader hints and warns when HTML has no plain-text fallback.
  • Mobile layout: warns when fixed widths over 640px are used without responsive rules/media queries.
  • Marketing compliance: if marketing terms are detected, checks that an unsubscribe path is present.
  • Spam-style signals: detects common spam phrases and excessive punctuation in the subject.
  • Empty content: flags messages with no readable HTML and no plain text.
How message content is prepared
  • The API endpoint GET /api/v2/messages/{id}/quality extracts the first text/html and text/plain parts from MIME content.
  • Body content is decoded when encoded as base64 or quoted-printable before checks run.
Packages used in this scoring path
  • MailHog-Server/emailquality (local scoring engine).
  • Go standard library in scoring: regexp, strings, html, math.
  • Go standard library in message decode path: encoding/base64, mime/quotedprintable, io/ioutil.
Customer conversation guide
  • Red result: explain that critical blockers were detected (for example unsafe HTML, no unsubscribe on marketing copy, or image-only content), which can directly reduce trust and inbox success.
  • Amber result: explain that the email can send, but quality risks were found that often lower engagement or increase filtering risk.
  • Green result with poor results: explain that content quality is good, then investigate sender-domain factors outside this checker.

Not currently checked here: SPF/DKIM/DMARC setup, sender/domain reputation, blocklists, complaint or bounce history, and mailbox-provider throttling.
